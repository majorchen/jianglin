# 《降临》完整技术方案与产品规划 v2

## 一、产品愿景

一个**每天自动更新的 2D 像素小镇**。思行社区的成员作为 AI agent 生活在这个世界里。每天早上打开网页，看到小镇发生了变化——新建筑出现了、角色移动了、故事推进了。底部投票决定明天的走向。

不是小说。不是游戏。是一个**活着的、会自己生长的世界。**

---

## 二、技术架构（三层分离）

```
┌─────────────────────────────────────────┐
│           Godot 游戏引擎                  │
│  2D 俯视地图 · 像素渲染 · 动画 · Web导出   │
├─────────────────────────────────────────┤
│           AI Agent 系统                   │
│  角色记忆 · 日常作息 · 社交关系 · 状态机    │
├─────────────────────────────────────────┤
│           Runner 内容管线                  │
│  剧情生成 · 图片生成 · 地图变化 · 投票处理    │
└─────────────────────────────────────────┘
```

### 2.1 Godot 引擎层（视觉与交互）

**选型理由：**
- MIT 开源协议，零授权费
- GDScript 类 Python 语法，Claude Code/Codex 可直接编写
- 内置 Tilemap 编辑器、AnimationPlayer、2D 物理引擎
- 一键导出 HTML5（Web 浏览器即玩）
- 场景文件 .tscn 为纯文本，AI agent 可直接修改

**职责：**
- 渲染 2D 俯视地图（像素风，256×256 瓦片）
- 管理角色精灵（行走动画、朝向、位置）
- 处理玩家交互（点击角色查看日记、滚动/缩放地图）
- 每日更新：加载 Runner 生成的新地图数据

### 2.2 AI Agent 系统（智能与生命感）

**参考：斯坦福小镇（Generative Agents, 2023）的算法思想**

每个 NPC 角色维护三个数据结构：

| 组件 | 说明 | 实现方式 |
|------|------|---------|
| **记忆流** (Memory Stream) | 角色经历的每件事，按时间排列 | JSON 文件，Runner 每天追加 |
| **日程表** (Daily Schedule) | 角色今天在什么时间做什么 | Runner 每天生成，Godot 执行 |
| **社交关系** (Social Graph) | 角色之间互相认识、互动记录 | 图结构，Runner 维护 |

**核心差异：不是实时 agent，是"剧本式 agent"。**
- 斯坦福小镇的 agent 是实时自主决策的——这需要持续运行的服务和大量计算
- 降临计划的 agent 是**每天生成一次剧本**——Runner 凌晨跑完，决定每个角色的当日行为
- Godot 加载剧本后，按时间推进播放角色动作
- 用户体验完全相同（看到角色在动），但技术成本降低 100 倍

**日程表示例：**
```json
{
  "character": "老王",
  "schedule": [
    {"time": "06:00", "action": "wake_up", "location": "shelter_3"},
    {"time": "07:00", "action": "repair_generator", "location": "power_station"},
    {"time": "12:00", "action": "eat", "location": "canteen"},
    {"time": "14:00", "action": "patrol", "location": "north_wall"},
    {"time": "18:00", "action": "talk_to", "target": "小李", "location": "canteen"}
  ]
}
```

### 2.3 Runner 内容管线（内容生产）

**已有基础：** 来信长安 Runner（已跑通 408 回合、64K 字、115 张图）

**新增输出：**
- 地图变化数据（JSON）：哪些建筑状态变了、解锁了哪些区域
- 角色日程表（JSON）：每个角色的当日行为脚本
- 对话快照（JSON）：关键对话内容
- 像素资产描述（Text）：新建筑/新角色的外观描述，供 AI 图片生成器使用

## 三、AI 辅助开发流水线

每个开发阶段都用 AI agent 执行，不是手写代码：

| 阶段 | 工具 | 做什么 |
|------|------|--------|
| 地图搭建 | Claude Code + Godot | 生成基础 Tilemap、建筑位置、NPC 站位点 |
| 角色系统 | Claude Code + Godot | 编写角色状态机、动画播放逻辑 |
| 地图更新 | Claude Code + Python | 读取 Runner JSON → 更新 Godot 场景文件 |
| 像素素材 | PixelLab / Sprite-AI | 根据文本描述生成角色精灵、建筑瓦片 |
| 部署 | Claude Code + Git | 构建 Web 版本 → push → Vercel 自动部署 |

## 四、四阶段实施路线

### Phase 1：静态小镇（1-2 周）

**目标：** 一张固定的像素地图，15 个 NPC 站在各自位置上，不可交互。

**交付物：**
- Godot 项目，一张 2D 俯视地图（废墟小镇主题）
- 15 个角色的像素精灵（AI 生成，静态站立图）
- 地图上标注角色名字
- 导出为 HTML5，部署到 Vercel
- 每天群里发链接：看，你们在这里

**验证指标：** 15 个群友是否点开看了？是否截图分享？

### Phase 2：活的地图（3-4 周）

**目标：** Runner 接入，地图每天自动变化。

**新增功能：**
- Runner 每天生成地图变化数据
- Godot 加载 JSON，更新建筑状态、角色位置
- 角色有简单的行走动画（在固定路线巡逻）
- 点击角色弹出今日状态卡片（文字+小图）
- 页面底部嵌入投票组件

**验证指标：** 用户是否每天回来？投票参与率是否 > 50%？

### Phase 3：社交小镇（5-8 周）

**目标：** 完整 agent 系统，角色有记忆、有社交。

**新增功能：**
- Runner 生成角色日程表，Godot 按时间播放
- 角色之间有对话（气泡文字）
- 角色记住过去发生的事（记忆流在对话中体现）
- 新成员加入 → 24 小时内新角色出现在地图上
- 投票结果直接影响第二天剧情和地图

**验证指标：** 周留存率 > 50%？是否有自然传播（非群主邀请的新读者）？

### Phase 4：理想形态（持续迭代）

- 更多地图区域（小镇→周边废土→其他幸存者据点）
- 季节性事件（暴风雪、沙尘暴、救援队到来）
- 第二季：新世界观（不限于 AI 末日）
- 开源核心引擎，让其他社群搭建自己的"活地图"

## 五、MVP 定义（Phase 1）

**一句话：** 一张像素地图，15 个群友的角色站在上面，每天打开能看到变化。

**最少可交付物：**
1. Godot 项目 ×1
2. 像素地图 ×1（128×128 瓦片）
3. 角色精灵 ×15（AI 生成）
4. Vercel 部署 ×1
5. Runner 扩展（生成地图变化 JSON）×1

**预估工时：** Claude Code + Godot，2-3 天搭建基础壳，1 周打磨。

**前置条件：** 15 个群友的标签数据（名字、身份、特征）。

## 六、与来信长安的分工

| | 来信长安 | 降临计划 |
|---|---|---|
| 定位 | 单人互动叙事游戏 | 社群可视化连载世界 |
| 引擎 | Next.js Web App | Godot 2D 游戏 |
| 内容 | 唐朝书信悬疑 | AI 末日群像剧 |
| 交互 | 选项/自由输入 | 投票驱动地图变化 |
| 技术积累 | API Runner (可复用) | Runner 扩展 + Godot 管线 |

**两者不冲突。** 来信长安验证了 Runner 的内容生产能力。降临计划用同一个 Runner 核，换了一个可视化前端。
